# 2026-03-20 存储策略记录

## 当前真实实现

- 当前系统的存储模型不是“给每个用户自动划一块独立磁盘空间”。
- 现在实际采用的是两层模型：
  - 物理层：一个共享存储后端，支持本地文件系统或 S3 兼容对象存储
  - 逻辑层：应用按用户的 `storageLimit` / `storageUsed` 做上传配额控制

## 当前已经具备的能力

- 管理台可以在 `local` 和 `s3` 之间切换存储后端
- 上传前会检查用户剩余存储配额
- 删除时会释放逻辑已用空间
- 相同内容文件支持物理存储复用，减少重复占用
- S3 模式下可在需要本地文件路径时临时拉取对象到本地缓存

## 当前没有做的事情

- 不会自动扩容本地磁盘
- 不会自动为每个用户分配独立 bucket / 独立目录容量上限
- 不会自动做冷热分层、生命周期归档、跨后端迁移
- 还没有容量阈值告警、备份巡检或对象生命周期治理

## 建议的未来方向

### 1. 部署口径

- 开发 / 单机：`local` 足够
- 生产：优先 `s3`，把物理容量问题交给对象存储平台处理

### 2. 应用层职责

- 继续负责：
  - 用户逻辑配额
  - 上传前校验
  - 后端切换
  - 物理对象复用

### 3. 基础设施层职责

- 逐步补：
  - bucket / 磁盘容量告警
  - 备份与恢复演练
  - 生命周期策略
  - 必要时的迁移脚本或冷数据归档

## 结论

- 当前“空间会自动规划吗”的答案是否定的。
- 现在的系统只会自动判断“这个用户还能不能继续上传”，不会自动解决底层物理容量怎么扩。
- 如果未来要把存储做稳，重点应该是 `S3-first + 监控告警 + 生命周期/备份`，而不是在应用里继续模拟物理分盘。
